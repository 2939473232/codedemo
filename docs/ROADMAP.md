# SpriteForge 开发优先级与实施路线

## 1. 排期原则

本路线图基于 `docs/PRD.md` 拆解，优先保证竞赛评审中最关键的三件事：

- 完整闭环：用户能创建项目、生成素材、管理素材、导出资源包。
- 工程质量：架构清晰、模块边界明确、PR 拆分合理、代码可维护。
- 演示稳定：即使真实图像生成服务不稳定，也能通过 fallback 数据完成流畅 demo。

优先级定义：

- P0：必须完成。缺少这些能力，作品无法形成完整 demo 或无法体现题目要求。
- P1：重要增强。能明显提升创新性、完成度和演示观感。
- P2：加分扩展。适合在 MVP 稳定后继续开发，不阻塞主线。

## 2. 总体优先级

### P0：端到端 MVP 闭环

目标：完成一个可运行、可演示、可导出的 2D 游戏素材生成工具。

| 优先级 | 模块 | 任务 | 价值 | 依赖 |
| --- | --- | --- | --- | --- |
| P0 | 项目初始化 | 初始化前后端项目、目录结构、代码规范、README | 建立工程基础，体现开发质量 | 无 |
| P0 | 项目与风格管理 | 创建项目、设置游戏类型、目标引擎、风格、视角、尺寸 | 支撑风格一致性和项目级资产包概念 | 项目初始化 |
| P0 | 生成表单 | 实现文本描述、素材类型、尺寸、数量、透明背景、用途等参数 | 满足“文本或简单参数生成素材”的核心题意 | 项目与风格管理 |
| P0 | Prompt 与参数 schema | 将表单参数转换为结构化生成请求，并做校验 | 保证生成链路稳定、可维护 | 生成表单 |
| P0 | 生成任务 API | 创建生成任务、查询任务状态、失败提示 | 打通前后端生成流程 | Prompt 与参数 schema |
| P0 | fallback 生成器 | 使用稳定示例素材或程序化素材模拟生成结果 | 保证演示稳定，降低模型接入风险 | 生成任务 API |
| P0 | 生成结果展示 | 网格展示候选素材，显示参数、尺寸、版本 | 让用户看到即时结果 | fallback 生成器 |
| P0 | 资产库 | 保存生成结果，支持按项目和类型查看 | 形成“资产管理”能力 | 生成结果展示 |
| P0 | 基础后处理 | 自动裁剪、尺寸规范化、透明背景标记或处理 | 体现游戏素材可用性 | 资产库 |
| P0 | 导出中心 | 导出 PNG、manifest.json、ZIP 资源包 | 完成接入游戏开发流程的关键闭环 | 资产库、基础后处理 |
| P0 | 演示项目模板 | 内置“森林冒险”示例项目和固定生成结果 | 保证 demo 清晰稳定 | fallback 生成器、资产库 |

### P1：作品亮点与评分增强

目标：让作品从“能用”变成“像一个真正的游戏素材工作台”。

| 优先级 | 模块 | 任务 | 价值 | 依赖 |
| --- | --- | --- | --- | --- |
| P1 | 角色动画 | 支持 idle/walk 动作帧生成和动画预览 | 强化 2D 游戏创作场景 | P0 生成链路 |
| P1 | Sprite Sheet | 合成角色动画 sprite sheet 和帧坐标 JSON | 明显体现引擎接入能力 | 角色动画 |
| P1 | Tile Set | 生成九宫格 tile set | 覆盖地图制作核心需求 | P0 生成链路 |
| P1 | Tile 拼接预览 | 在 Canvas 中预览 tile 拼接效果 | 直观展示素材质量 | Tile Set |
| P1 | 批量图标生成 | 根据多行输入生成道具/技能图标 | 提升效率和实用性 | P0 生成链路 |
| P1 | 风格一致性增强 | 项目级 prompt 模板、调色板约束、负向提示词 | 对应题目中的风格一致性要求 | Prompt 与参数 schema |
| P1 | Godot/Unity 目录模板 | 导出时生成引擎友好的目录结构 | 提升“无缝融入流程”的说服力 | 导出中心 |
| P1 | 错误处理与 loading | 完整空状态、加载态、失败重试、提示信息 | 提升交互流畅度与健壮性 | P0 页面与 API |
| P1 | 基础测试 | 对 schema、导出 manifest、后处理工具写测试 | 支撑开发质量评分 | P0 核心模块 |

### P2：高级能力与后续扩展

目标：在时间允许时增加创新性和长期价值。

| 优先级 | 模块 | 任务 | 价值 | 依赖 |
| --- | --- | --- | --- | --- |
| P2 | 真实生成 API | 接入图像生成 API 或 ComfyUI 工作流 | 提升真实生成能力 | P0 fallback 生成链路 |
| P2 | 参考图上传 | 用户上传风格参考图或角色参考图 | 增强可控性 | 真实生成 API |
| P2 | 局部重生成 | 对单个素材局部擦除和重绘 | 提升编辑能力 | 基础后处理 |
| P2 | 调色板映射 | 将素材颜色映射到项目调色板 | 提升风格一致性 | 风格一致性增强 |
| P2 | Tiled 兼容导出 | 导出 Tiled tileset JSON | 扩大工具链兼容范围 | Tile Set |
| P2 | Godot 插件原型 | 在 Godot 中一键导入导出包 | 强化工程接入演示 | Godot 目录模板 |
| P2 | 团队协作与审稿 | 素材评论、版本对比、审核状态 | 适合后续产品化 | 资产库 |

## 3. 推荐开发阶段

### 阶段 0：仓库与规范准备

目标：让项目可运行、可提交、可协作。

建议任务：

- 初始化应用框架。
- 配置 TypeScript、lint、format。
- 编写 README，说明项目目标、启动方式和开发约定。
- 建立 `docs/`、`frontend/`、`backend/` 或单体应用目录。

完成标准：

- 本地可以一键启动。
- README 能让新成员理解如何运行。
- 第一个工程 PR 可以独立审查。

建议 PR：

- PR 1：初始化项目工程与开发规范。

### 阶段 1：产品主流程页面

目标：搭出用户能理解的产品骨架。

建议任务：

- 项目创建页。
- 项目工作台布局。
- 左侧项目/资产导航。
- 中央生成面板。
- 右侧参数或结果详情面板。
- 示例项目入口。

完成标准：

- 用户能创建或打开一个项目。
- 用户能看到生成表单和素材结果区域。
- 页面状态清楚，无明显交互断点。

建议 PR：

- PR 2：实现项目创建与工作台布局。
- PR 3：实现素材生成表单和参数配置。

### 阶段 2：生成闭环

目标：从输入到结果展示完整跑通。

建议任务：

- 定义 Project、GenerationRequest、Asset 类型。
- 实现参数 schema 校验。
- 实现 prompt service。
- 实现生成任务 API。
- 实现 fallback 生成器。
- 前端轮询或模拟任务进度。
- 展示生成结果网格。

完成标准：

- 用户输入描述后可以获得候选素材。
- 生成中、成功、失败状态清晰。
- 生成结果保存基础参数和版本信息。

建议 PR：

- PR 4：实现核心数据模型与参数 schema。
- PR 5：实现生成任务 API 与 fallback 生成器。
- PR 6：实现生成结果展示与任务状态。

### 阶段 3：资产库与后处理

目标：把生成结果变成可管理资产。

建议任务：

- 资产列表。
- 资产详情。
- 类型筛选。
- 收藏。
- 删除。
- 复制生成参数。
- 基础后处理：裁剪、缩放、透明背景处理或标记。

完成标准：

- 用户能找到历史生成素材。
- 用户能筛选角色、道具、图标、tile。
- 用户能对素材做基础规范化处理。

建议 PR：

- PR 7：实现资产库、详情页和筛选。
- PR 8：实现基础后处理能力。

### 阶段 4：导出与游戏开发接入

目标：体现本项目与普通生图工具的差异。

建议任务：

- 导出单图 PNG。
- 批量导出 ZIP。
- 生成 manifest.json。
- 生成 Unity 或 Godot 友好目录结构。
- 导出结果预览。

完成标准：

- 用户可以选择一个项目或一批素材导出。
- ZIP 中包含图片和 metadata。
- 导出目录命名清晰，可解释如何导入引擎。

建议 PR：

- PR 9：实现 manifest 与 ZIP 导出。
- PR 10：实现 Godot/Unity 目录模板。

### 阶段 5：亮点功能

目标：增加评审记忆点。

建议任务：

- 角色 walk 动画帧生成。
- 动画播放预览。
- sprite sheet 合成。
- tile set 生成。
- tile 拼接预览。
- 批量图标生成。

完成标准：

- 至少一个角色动画能在页面播放。
- 至少一套 tile 能展示拼接预览。
- 批量道具图标能统一风格展示。

建议 PR：

- PR 11：实现角色动画预览与 sprite sheet。
- PR 12：实现 tile set 和拼接预览。
- PR 13：实现批量图标生成。

### 阶段 6：演示与质量收尾

目标：提升作品表达和稳定性。

建议任务：

- 演示脚本。
- 固定 demo 数据。
- 错误处理。
- loading 状态。
- 核心测试。
- 部署说明。
- 录屏大纲。

完成标准：

- 3 分钟内可以完整演示从项目创建到导出。
- 失败场景有提示，不会卡死。
- README 和 demo 文档可支撑评审理解。

建议 PR：

- PR 14：补充错误处理、loading 和空状态。
- PR 15：补充核心测试。
- PR 16：补充 demo 脚本、部署说明和录屏大纲。

## 4. 第一版 MVP 必须交付清单

第一版不要追求功能多，优先做出完整体验。

必须交付：

- Web 应用可以启动。
- 有项目创建和风格配置。
- 有素材生成表单。
- 支持角色、道具、图标、tile 四种类型。
- 有稳定 fallback 生成结果。
- 有生成结果网格。
- 有资产库和类型筛选。
- 有基础后处理。
- 有 ZIP 导出和 manifest.json。
- 有森林冒险演示项目。
- 有 README 和 demo 操作文档。

可以暂缓：

- 真实 AI 生成 API。
- 复杂动画动作。
- 高级图像编辑。
- 完整 Godot/Unity 插件。
- 用户登录和云端存储。

## 5. 关键依赖关系

```text
项目初始化
  -> 项目与风格管理
    -> 生成表单
      -> 参数 schema / prompt service
        -> 生成任务 API
          -> fallback 生成器
            -> 生成结果展示
              -> 资产库
                -> 后处理
                  -> 导出中心
                    -> 演示项目与录屏
```

亮点功能依赖：

```text
生成任务 API + 资产库
  -> 角色动画
    -> sprite sheet 导出

生成任务 API + 资产库
  -> tile set
    -> tile 拼接预览
      -> Tiled/Godot metadata

生成表单 + fallback 生成器
  -> 批量图标生成
```

## 6. 评分导向的工作重点

### 作品完整度与创新性

优先展示：

- 项目级风格管理。
- 文本加参数生成。
- 资产库。
- 动画或 tile 预览。
- 导出资源包。

不要把演示重点放在单张图片是否足够惊艳，而要强调它能进入游戏开发流程。

### 开发过程与质量

优先保证：

- PR 小而清楚。
- 每个 PR 有明确目标。
- commit 信息规范。
- schema、导出、后处理等关键逻辑有测试。
- 代码模块边界清晰。

建议 commit 风格：

```text
docs: add roadmap
feat: scaffold web app
feat: add project workspace
feat: add generation request schema
feat: implement fallback generator
feat: add asset library
feat: add zip export
test: cover export manifest generation
docs: add demo script
```

### 演示与表达

优先准备：

- 一个固定主题：森林冒险 Roguelike。
- 一条固定路径：创建项目 -> 生成角色 -> 生成道具 -> 生成 tile -> 预览 -> 导出。
- 一份导出资源包。
- 一段 3 分钟 demo 视频。

## 7. 近期建议执行顺序

当前仓库已经有 PRD，下一步建议按以下顺序推进：

1. 初始化项目工程。
2. 完成 README 和本地启动说明。
3. 完成项目创建与工作台布局。
4. 完成生成表单和参数 schema。
5. 完成 fallback 生成链路。
6. 完成资产库。
7. 完成 ZIP 导出。
8. 再做动画、tile、批量图标等亮点功能。

如果时间紧，只做到第 7 步也能形成完整作品；如果时间充足，第 8 步会明显提高创新性和演示效果。
