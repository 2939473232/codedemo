# 2D 游戏素材生成工具 PRD

## 1. 项目概述

### 1.1 项目名称

暂定名称：SpriteForge

### 1.2 项目定位

SpriteForge 是一个面向独立游戏开发者、游戏美术、游戏策划和学生团队的 2D 游戏素材生成工具。用户通过输入文本描述、选择游戏类型、素材类别、画面风格、尺寸规格和导出参数，可以快速生成可直接用于 2D 游戏开发流程的素材包。

工具重点解决 2D 游戏创作中的三个痛点：

- 素材制作成本高：小团队缺少稳定美术产能，外包或手绘成本较高。
- 风格统一困难：不同批次、不同角色、不同道具之间容易风格不一致。
- 接入开发流程麻烦：生成图片如果缺少透明背景、切图、命名、元数据和导出规范，很难直接用于 Unity、Godot、Cocos Creator、RPG Maker 等工具。

### 1.3 项目目标

在竞赛周期内完成一个可演示、可扩展、可落地的 2D 游戏素材生成工具原型。核心目标包括：

- 允许用户通过文本和简单参数生成 2D 游戏素材。
- 支持常见游戏素材类别，包括角色、敌人、道具、图标、地块、背景、UI 元素等。
- 支持统一风格控制，让同一项目下的素材具有一致的视觉表现。
- 支持游戏开发友好的导出格式，包括 PNG、透明背景、精灵图、图标包、瓦片图集和元数据。
- 提供清晰流畅的交互流程，方便评审快速理解产品价值。
- 形成合理的工程架构、提交记录和演示材料。

## 2. 背景与问题分析

### 2.1 2D 游戏素材常见需求

常规 2D 游戏开发对素材的需求通常包括：

- 角色素材：主角、NPC、敌人、Boss、皮肤变体。
- 动画帧：待机、行走、攻击、受击、死亡、跳跃、施法等。
- 道具与装备：武器、防具、消耗品、任务物品、掉落物。
- 图标素材：技能图标、背包图标、状态图标、成就图标。
- 场景素材：背景图、前景装饰、建筑、自然元素。
- 地块素材：平台、墙面、地面、水面、道路、草地、岩石等 tile。
- UI 素材：按钮、面板、血条、经验条、弹窗、边框、货币图标。
- 特效素材：爆炸、光效、攻击轨迹、魔法阵、粒子序列。

这些素材在游戏开发中并不是单张图片即可满足，还需要满足工程使用要求：

- 尺寸符合引擎导入规范，例如 16x16、32x32、64x64、128x128。
- 背景透明，边缘干净。
- 命名规范清晰，方便程序引用。
- 同一动画动作的帧数一致，便于播放。
- tile 能够无缝拼接。
- 图集或精灵表带有坐标元数据。
- 同一项目中的角色、道具、UI 和场景风格一致。

### 2.2 目标用户

#### 独立游戏开发者

通常具备编程能力，但缺少稳定美术产能。需要快速产出可用素材，用于原型开发、Game Jam、小型商业项目或课程设计。

核心需求：

- 快速生成角色、道具、图标和地图素材。
- 能够直接导入游戏引擎。
- 低成本迭代多个版本。

#### 游戏美术与概念设计师

具备美术判断能力，需要更快地产生草案、变体和批量素材。

核心需求：

- 根据风格板生成同一系列素材。
- 对已有素材进行扩展、变体和补全。
- 批量导出并交给程序或关卡设计使用。

#### 游戏策划与学生团队

需要表达玩法和视觉方向，但不一定有专业绘制能力。

核心需求：

- 通过自然语言描述快速生成 demo 素材。
- 用统一视觉风格完成演示。
- 降低从想法到可运行原型的门槛。

### 2.3 核心问题

本项目需要解决的不只是“生成一张图”，而是“生成能进入游戏制作流程的素材”。核心问题包括：

- 需求表达问题：用户不一定知道该如何写高质量提示词。
- 风格一致问题：一次生成的素材可用，但多次生成容易风格漂移。
- 素材规范问题：生成图像可能不符合透明背景、尺寸、帧序列、tile 拼接等游戏需求。
- 资产管理问题：多个素材需要按项目、类型、风格、版本组织。
- 导出接入问题：需要适配主流游戏引擎和开发流程。

## 3. 产品方案

### 3.1 产品形态

优先开发 Web 应用原型，后续可扩展为桌面端或引擎插件。

推荐原因：

- Web 端便于演示和评审。
- 无需用户安装复杂环境。
- 前后端架构清晰，适合展示工程质量。
- 可以方便接入图像生成 API、本地模型或模拟生成服务。

### 3.2 核心设计理念

#### 从游戏资产包出发，而不是单图生成

用户创建的是一个游戏项目资产包，系统围绕该项目维护统一风格、尺寸规范、命名规则和导出配置。

#### 从参数化需求出发，降低提示词门槛

用户不需要完整掌握提示词工程。界面提供素材类型、游戏视角、风格、尺寸、动作、数量、颜色倾向等结构化参数，系统自动组合成高质量生成请求。

#### 从开发流程出发，提供可用导出

生成结果需要面向游戏引擎可用，而不是只停留在展示图。导出文件应包含图片、元数据和使用说明。

## 4. 功能需求

### 4.1 项目与风格管理

#### 功能描述

用户可以创建一个素材项目，并为项目设置全局风格。

#### 关键能力

- 创建项目：项目名称、游戏类型、目标引擎、默认分辨率。
- 设置风格：像素风、手绘卡通、暗黑奇幻、赛博朋克、Q 版、低饱和童话等。
- 设置视角：俯视角、侧视角、等距视角、正面图标。
- 设置调色板：主色、辅助色、禁用色、明暗倾向。
- 保存风格预设：后续生成默认继承项目风格。

#### 验收标准

- 用户可以创建项目并保存风格配置。
- 同一项目下生成表单自动继承风格信息。
- 风格配置可以编辑并影响后续生成。

### 4.2 文本与参数化生成

#### 功能描述

用户输入素材描述，并选择简单参数，系统生成一组候选素材。

#### 支持参数

- 素材类型：角色、敌人、道具、图标、tile、背景、UI、特效。
- 文本描述：例如“戴红色围巾的猫骑士，使用短剑”。
- 风格：继承项目风格，也可单次覆盖。
- 尺寸：16x16、32x32、64x64、128x128、256x256。
- 生成数量：1、2、4、8。
- 透明背景：开启或关闭。
- 轮廓强度：无、弱、中、强。
- 色彩限制：不限、项目调色板、黑白剪影。
- 用途：游戏内角色、背包图标、商店图标、地图装饰等。

#### 生成结果

- 多张候选图。
- 每张图显示尺寸、生成参数、版本号。
- 支持收藏、重生成、复制参数、下载单图。

#### 验收标准

- 用户可以通过文本加参数生成素材。
- 生成结果按网格展示。
- 生成记录可回溯。

### 4.3 角色与动画帧生成

#### 功能描述

用户可以生成角色基础图，并进一步生成动作帧序列。

#### 支持动作

- idle：待机
- walk：行走
- run：奔跑
- attack：攻击
- hurt：受击
- death：死亡
- jump：跳跃

#### 关键能力

- 指定动作和帧数。
- 保持角色外观一致。
- 预览动画播放。
- 导出为单帧 PNG 或 sprite sheet。
- 生成 metadata.json，记录帧坐标、动作名、帧率。

#### 验收标准

- 能选择角色素材生成动画帧。
- 页面中可播放动画预览。
- 可导出包含图片和元数据的动画资源包。

### 4.4 Tile 与地图素材生成

#### 功能描述

用户可以生成用于 2D 地图搭建的 tile set。

#### 支持类型

- 地面 tile：草地、泥土、石板、雪地、沙地。
- 边缘 tile：墙角、平台边缘、悬崖边。
- 装饰 tile：花草、石头、箱子、树桩。
- 水体 tile：静态水面、河流边缘。

#### 关键能力

- 选择 tile 尺寸：16x16、32x32、48x48、64x64。
- 支持九宫格 tile set。
- 支持无缝拼接检测。
- 支持导出 tile sheet。
- 可生成 Godot TileSet 或 Tiled map editor 兼容元数据。

#### 验收标准

- 用户可以生成一组 tile。
- 能在页面预览拼接效果。
- 导出文件可被常见地图编辑工具使用。

### 4.5 图标与道具批量生成

#### 功能描述

用户可以批量生成游戏图标和道具素材，用于背包、技能栏、商店等系统。

#### 关键能力

- 输入列表批量生成，例如“铁剑、木盾、治疗药水、火球术”。
- 统一图标边框、光照和背景规则。
- 支持透明背景或统一底板。
- 支持按名称导出。

#### 验收标准

- 支持多条素材描述批量输入。
- 生成结果保持统一尺寸和风格。
- 导出文件名与输入名称对应。

### 4.6 素材编辑与后处理

#### 功能描述

对生成素材进行轻量编辑，使其更符合游戏开发要求。

#### 关键能力

- 背景移除或透明化。
- 自动裁剪透明边距。
- 尺寸缩放与像素对齐。
- 边缘描边。
- 调色板映射。
- 简单擦除和局部重生成。
- 镜像、旋转、留白修正。

#### 验收标准

- 用户可以对生成素材进行基础后处理。
- 后处理结果可另存为新版本。

### 4.7 资产库与版本管理

#### 功能描述

系统保存用户生成过的素材，并按项目、类型和标签组织。

#### 关键能力

- 素材列表和详情页。
- 标签、收藏、搜索。
- 版本记录。
- 参数复用。
- 批量选择和批量导出。

#### 验收标准

- 用户能查看历史生成记录。
- 用户能筛选指定类型素材。
- 用户能从旧版本继续生成变体。

### 4.8 导出与引擎接入

#### 功能描述

用户可以将素材导出为主流 2D 游戏开发流程可用的资源包。

#### 支持导出

- PNG：单图或批量图片。
- Sprite Sheet：角色动画、特效序列、图标合集。
- JSON：通用帧坐标和素材元数据。
- TexturePacker 风格 JSON：便于引擎读取。
- Godot 导入包：目录结构和 metadata。
- Unity 友好目录：Sprites、Animations、Tiles、UI。
- Tiled 兼容 tileset JSON。

#### 推荐目录结构

```text
export/
  sprites/
    hero/
      hero_idle.png
      hero_walk.png
      hero_spritesheet.png
      hero_spritesheet.json
  tilesets/
    forest_tileset.png
    forest_tileset.json
  icons/
    iron_sword.png
    health_potion.png
  ui/
    button_primary.png
  manifest.json
```

#### 验收标准

- 用户可以选择导出格式。
- 导出的压缩包包含图片和元数据。
- 导出命名规范清晰。

## 5. 非功能需求

### 5.1 生成效率

- 单次生成应给出明显进度状态。
- 支持异步任务队列，避免页面卡死。
- 支持批量任务并发限制。
- 对重复参数可缓存结果。

### 5.2 素材质量

- 尽量保证透明背景、边缘干净、主体完整。
- 对 pixel art 类素材应避免模糊和半透明脏边。
- 生成后提供质量检查提示，例如尺寸异常、背景未透明、边缘溢出。

### 5.3 风格一致性

- 项目级风格预设必须参与每次生成。
- 支持保存参考图或风格种子。
- 同一素材系列复用基础提示词和负向约束。

### 5.4 可维护性

- 前后端职责清晰。
- 生成服务与业务服务解耦。
- 导出逻辑模块化，便于扩展 Unity、Godot、Cocos 等目标。
- 关键参数使用类型定义和 schema 校验。

### 5.5 安全与合规

- 用户输入需要过滤明显违规内容。
- 生成记录应和项目隔离。
- 如果接入第三方生成 API，应隐藏密钥，不在前端暴露。

## 6. 用户流程

### 6.1 首次使用流程

1. 用户进入应用首页。
2. 创建新项目，输入项目名称和游戏类型。
3. 选择目标引擎，例如 Unity 或 Godot。
4. 选择整体风格，例如 32x32 像素风、温暖奇幻、俯视角。
5. 进入素材工作台。
6. 选择素材类型并输入描述。
7. 生成候选素材。
8. 收藏满意结果或进行重生成。
9. 对素材进行裁剪、描边、透明化等后处理。
10. 导出素材包。

### 6.2 演示核心流程

演示建议围绕一个完整小游戏主题展开，例如“森林冒险 Roguelike”。

1. 创建项目：森林冒险，Godot，32x32 像素风，俯视角。
2. 生成主角：戴红色围巾的小骑士。
3. 生成敌人：蘑菇怪、史莱姆、森林幽灵。
4. 生成道具：铁剑、治疗药水、金币、钥匙。
5. 生成 tile set：草地、泥土路、石墙、树丛。
6. 生成主角 walk 动画帧。
7. 预览动画和 tile 拼接效果。
8. 导出 Godot 资源包。
9. 展示导出目录和 metadata。

## 7. 系统架构方案

### 7.1 推荐技术栈

#### 前端

- React 或 Vue：构建交互界面。
- TypeScript：提升类型安全和可维护性。
- Canvas API：实现图片预览、裁剪、拼接预览、动画播放。
- Zustand 或 Pinia：管理项目配置、生成状态和资产库状态。
- Tailwind CSS 或 UnoCSS：快速实现一致 UI。

#### 后端

- Node.js + NestJS / Express，或 Python + FastAPI。
- REST API 或轻量 RPC。
- 任务队列：BullMQ、Celery 或内存队列原型。
- 文件存储：本地文件系统，后续可扩展到对象存储。
- 数据库：SQLite 作为原型数据库，后续可迁移 PostgreSQL。

#### 生成服务

根据比赛环境和成本选择三档方案：

- Demo 模拟生成：使用预置素材、程序化占位图和可重复生成结果，保证演示稳定。
- API 生成：接入图像生成 API，展示真实文本到图像能力。
- 本地模型：接入 Stable Diffusion / ComfyUI 工作流，强调低成本和可控性。

建议比赛原型采用“真实生成接口 + 稳定 fallback”的设计，避免演示现场因网络或模型波动失败。

### 7.2 模块划分

```text
frontend/
  project-workspace/
  asset-generator/
  asset-library/
  asset-editor/
  export-center/

backend/
  project-service/
  prompt-service/
  generation-service/
  postprocess-service/
  asset-service/
  export-service/
```

### 7.3 后端核心模块

#### Project Service

负责项目创建、项目配置、风格预设和目标引擎配置。

#### Prompt Service

负责把用户的自然语言和结构化参数转换为生成模型可理解的提示词。

核心能力：

- Prompt 模板管理。
- 素材类型约束。
- 风格一致性约束。
- 负向提示词生成。
- 参数 schema 校验。

#### Generation Service

负责调用生成模型或 fallback 生成器。

核心能力：

- 创建生成任务。
- 查询任务状态。
- 管理失败重试。
- 保存原始输出。

#### Postprocess Service

负责透明背景、裁剪、缩放、描边、sprite sheet 合成、tile 拼接检查等。

#### Asset Service

负责素材记录、版本、标签、收藏和文件路径管理。

#### Export Service

负责按目标引擎和导出格式生成资源包。

## 10. MVP 范围

### 10.1 必须完成

- 项目创建与风格配置。
- 文本加参数生成素材。

## 13. 风险与应对

### 13.1 图像生成不稳定

风险：真实生成模型可能输出质量波动，影响演示。

应对：

- 准备 fallback 示例素材。
- 对演示项目使用固定 seed 或固定结果。
- 生成任务失败时提供重试和示例替代。

### 13.2 风格一致性不足

风险：同一项目下的素材看起来不像同一套。

应对：

- 使用项目级 prompt 模板。
- 保存风格参数和参考图。
- 限制调色板和尺寸。
- 对同系列素材复用基础描述。

### 13.3 导出格式复杂

风险：不同引擎导入规则不同，短期内难以完全适配。

应对：

- MVP 优先支持通用 PNG、sprite sheet、JSON、ZIP。
- 针对 Godot 或 Unity 选择一个重点优化。
- 其他引擎以目录模板和 manifest 支持。

### 13.4 开发周期紧张

风险：功能范围过大。

应对：

- 先完成端到端 MVP，再逐步增强。
- 保持 fallback 生成器，避免生成模型接入阻塞主流程。
- 将高级编辑、多人协作等功能延后。

## 14. 成功指标

### 14.1 产品指标

- 用户能在 5 分钟内创建一个项目并生成第一批素材。
- 用户能在 10 分钟内导出一个可用于游戏原型的素材包。
- 至少支持 4 类素材生成。
- 同项目素材在视觉风格上有明显一致性。

### 14.2 工程指标

- 应用可本地一键启动。
- 核心接口有基础测试。
- 导出资源包结构稳定。
- PR 和 commit 记录能体现合理开发过程。

### 14.3 演示指标

- demo 视频能完整展示输入、生成、预览、导出流程。
- 能清楚解释工具如何融入 2D 游戏开发流程。
- 能展示至少一个导出资源包和一个引擎接入示例。
